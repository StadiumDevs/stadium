# Testing email & admin/judge sign-in

Stadium uses **two independent email systems**. Don't conflate them — they're
configured and tested separately.

| Purpose | Sent by | Configured by | How to test |
|---|---|---|---|
| Invite / notification emails | **Resend** | `RESEND_API_KEY` + `RESEND_FROM_EMAIL` (server / Railway) | `npm run verify:email` |
| **Magic-link sign-in email** | **Supabase Auth** (its own provider) | Supabase dashboard → Authentication → Email | sign in with a real inbox |

The invite email only contains a link to `/admin/programs/:slug`. The actual
sign-in link is generated by **Supabase Auth** when the person enters their email
on that page (`supabase.auth.signInWithOtp`, `client/src/lib/auth/useSocialAuth.ts`).

---

## A. Resend — invite / notification email

**Config (Railway server env):** `RESEND_API_KEY`, and `RESEND_FROM_EMAIL` on a
domain **verified in Resend**. If `RESEND_API_KEY` is unset, every send is recorded
as `provider_not_configured` and no email goes out.

**Fastest check — one command, no admin auth, no DB:**

```bash
cd server
# Put the deploy's RESEND_API_KEY / RESEND_FROM_EMAIL in server/.env to validate
# the real key + verified domain, then:
npm run verify:email -- you@example.com
```

- Success → prints the Resend message id; the email arrives in the inbox.
- `RESEND_API_KEY not set` → key missing in this env.
- `Send failed` → usually `RESEND_FROM_EMAIL` is not on a verified domain, or a bad recipient.

**Is the deployed env wired?** (without sending anything)

```bash
curl https://<railway-host>/api/health
# → { ..., "email": { "resendConfigured": true, "fromConfigured": true } }
```

**In-app check + audit trail:** as a wallet admin on `/admin/programs/:slug`,
invite your own email → the response carries `emailSent: true`. Cross-check:
- the `notifications` table (`status='sent'`, `provider_message_id` set), and
- the Resend dashboard logs (delivery / bounce).

## B. Supabase Auth — the magic-link sign-in

This is what an invited admin/judge uses to log in. It does **not** go through Resend.

**Config:**
- Client (Vercel, Preview **and** Prod): `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` — same Supabase project as the server's `SUPABASE_URL`. If unset, the UI shows "email sign-in not configured".
- Server (Railway): `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` — used to verify the token (`server/api/auth/supabaseUser.js`).
- **Supabase dashboard → Authentication → URL Configuration → Redirect URLs:** add the exact deployed origin(s) (e.g. `https://stadium.joinwebzero.com`, plus any preview origin you test on). Sign-in uses `emailRedirectTo = window.location.href` (the `/admin/programs/:slug` page); if the origin isn't allowlisted, the link errors.

> **Test on a NON-mock deployment.** A preview built with `VITE_USE_MOCK_DATA=true`
> never calls the server, so it can't exercise real sign-in or the server-side
> authorization gates. Test on prod, or on a preview built with
> `VITE_USE_MOCK_DATA=false` and `VITE_API_BASE_URL` pointing at the real Railway API.

**Steps:**
1. As a wallet admin on `/admin/programs/:slug`, invite your email — role **judge** or **admin**.
2. Sign out (or use an incognito window) and open `/admin/programs/:slug`.
3. Enter that email → **EMAIL ME A LINK**.
4. Open the **Supabase** magic-link email in your inbox; it redirects back to the page.
5. Confirm the role-correct surface:
   - **judge** → sees only the scoring section (no applications, signups, inbox, or admin roster);
   - **admin** → sees a `VIEW-ONLY` badge + the applications list.
   - Both are confined to that one event; another event's URL shows "not authorized".

---

## C. Scale callout for the event (~100 sign-ins)

Supabase's **built-in** auth email is rate-limited to a few messages per hour —
fine for a smoke test, **not** for inviting ~100 judges/winners. Before the event,
configure a **custom SMTP provider in Supabase → Authentication → Email** (e.g.
Resend SMTP) and raise the rate limit. This is a dashboard change, not code, and
should be done well ahead of time.
